# ADR-001: 双库合并可行性评估

## 状态

已接受 — 方案 A（合并为单库）已于 2026-06-23 执行完成

## 背景

nav-site 当前采用双库架构：开发库（`nzaocqwumlmbewoddysd`）作为唯一写入源，生产库（`vyqqbypwrbdcafanzwmj`）作为只读副本，通过 GitHub Actions 每 6 小时同步。

### 双库架构带来的成本

1. **同步延迟**：管理员修改后最长 6 小时才在生产环境生效
2. **两套同步逻辑**：`sync-db.mjs`（脚本）和 `sync/route.ts`（API）逻辑不一致
3. **环境变量复杂度**：需要维护 4 组 Supabase 配置（DEV URL/KEY + PROD URL/KEY）
4. **数据一致性风险**：源库故障可能导致误删生产数据（已添加保护但仍存在风险）
5. **CI/CD 复杂度**：sync-db.yml 需要额外的 secrets 配置
6. **开发体验**：本地开发需要配置双库环境变量

### 双库架构带来的收益

1. **读写隔离**：管理操作不影响公开页面性能
2. **数据安全**：开发库的误操作不会直接影响生产环境
3. **回滚能力**：生产库可作为快照回滚到上次同步状态

## 决策驱动因素

| 因素 | 当前值 | 阈值 |
|------|--------|------|
| 日均写入次数 | < 10 次 | > 1000 次/天时双库有价值 |
| 日均读取次数 | ISR 缓存覆盖 95%+ | > 10000 次/天时考虑读写分离 |
| 管理员数量 | 1 人 | > 5 人时写入冲突概率增加 |
| 同步延迟容忍度 | 6 小时 | < 5 分钟时双库不可接受 |
| 数据一致性要求 | 中 | 高一致性要求时双库是负担 |

## 方案对比

### 方案 A：合并为单库（推荐）

**做法**：废弃开发库，所有读写操作直接连生产库。依赖 Supabase RLS 策略保证数据安全。

**优势**：
- 消除同步延迟（修改立即生效）
- 消除两套同步逻辑的维护成本
- 环境变量减半
- CI/CD 配置简化
- 数据一致性保证

**劣势**：
- 失去读写隔离（但 ISR 缓存已覆盖大部分读取）
- 误操作直接影响生产（但 RLS 策略可限制 anon 用户写入）

**迁移步骤**：
1. 审计生产库 RLS 策略，确保 anon 用户只能写入 `submit_attempts`、`login_attempts`、`click_rate_limits` 表
2. `nav_links` 表的写入仅允许认证用户（admin role）
3. 修改 `lib/supabase/config.ts`，移除双库路由逻辑
4. 修改 `lib/supabase/admin.ts`，改为使用与 `server.ts` 相同的客户端
5. 删除 `scripts/sync-db.mjs` 和 `app/api/sync/route.ts`
6. 删除 `.github/workflows/sync-db.yml`
7. 移除 `SOURCE_SUPABASE_*` 和 `*_DEV` 环境变量

**风险**：RLS 策略配置不当可能导致数据泄露或未授权写入。迁移前必须完成 RLS 审计。

### 方案 B：保留双库但修复同步逻辑

**做法**：保留双库架构，但统一同步逻辑（将 `sync-db.mjs` 替换为调用 `sync/route.ts` API）。

**优势**：
- 保留读写隔离
- 同步逻辑统一
- 迁移成本低

**劣势**：
- 同步延迟仍然存在
- 环境变量复杂度不变
- 维护成本仍然高

### 方案 C：保留双库但缩短同步间隔

**做法**：将同步间隔从 6 小时缩短到 30 分钟。

**优势**：延迟降低到可接受范围

**劣势**：
- GitHub Actions 使用量增加（免费额度 2000 分钟/月）
- 全量同步频率增加，数据库压力增大
- 核心问题（两套同步逻辑、环境变量复杂度）未解决

## 建议

**推荐方案 A（合并为单库）**，前提是完成 RLS 策略审计。

理由：
1. 当前流量级别下，ISR 60 秒缓存已覆盖 95%+ 的读取请求，读写隔离的价值有限
2. 双库架构的维护成本（6 项）远超其收益（3 项）
3. 合并后管理员修改立即生效，用户体验显著提升
4. RLS 策略 + ISR 缓存足以保证数据安全和性能

**决策截止日期**：Q1 结束前（2026-09-30）
